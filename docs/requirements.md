# 要件定義（Requirements）

## 目的

AI エージェントオーケストレーションの最新手法を調査・考案・研究・検証・実証・実装し、dev-orchestration-template および他のプロジェクトリポジトリに継続して適用可能な知見と成果を蓄積する。

## スコープ

### 対象（本リポジトリで扱う）

- エージェント並列実行の研究・検証・実装
- 型安全オーケストレーション（LangGraph + Pydantic AI）の研究
- パイプライン品質評価（Eval フレームワーク）の研究
- A2A（Agent2Agent）プロトコルの実装検証
- AI Gateway（LiteLLM 等）の統合検証
- TDD 並列フローの研究
- 上記成果の dev-orchestration-template へのフィードバック

### 非対象（本リポジトリでは行わない）

- 本番プロダクトの開発
- プロジェクト固有のビジネスロジック実装
- 外部 API の本番接続

## 用語

| 用語 | 定義 |
|---|---|
| 正本（SSOT） | Single Source of Truth。docs/ 配下の文書を唯一の信頼源とする |
| 受入条件（AC） | Issue / PR が完了と判断できる基準 |
| 実験コード | experiments/ 配下のプロトタイプコード。品質基準は緩和される |
| 昇格 | 実験コードが検証を経て src/ に移動すること |
| フィードバック | 研究成果を dev-orchestration-template に反映すること |

## 成功条件（ゴール）

- エージェント並列実行により実装品質が向上することを定量的に検証できる
- 研究成果が dev-orchestration-template にフィードバック可能な形で文書化されている
- CI が品質ゲートとして機能し、失敗時はマージできない
- 各研究テーマの知見が docs/research/ に蓄積されている

## 機能要件（FR）

### FR-001 エージェント並列実行フレームワーク

- 入力：タスク定義、エージェント設定、並列実行パラメータ
- 出力：各エージェントの実行結果、品質メトリクス、パフォーマンス比較
- 失敗時：フェイルクローズ（個別エージェントの失敗を隔離し、全体を停止しない）
- 担当モジュール：`src/orchestration_lab/parallel/`

### FR-010 パイプライン品質評価フレームワーク

- 入力：パイプライン実行ログ、エージェント応答
- 出力：品質スコア（CI 通過率、指摘数、修正回数等）
- 担当モジュール：`src/orchestration_lab/eval/`

### FR-020 実験管理

- 入力：実験パラメータ、設定ファイル
- 出力：再現可能な実験結果、比較レポート
- 担当モジュール：`experiments/`

## 非機能要件（NFR）

### NFR-001 再現性

- 実験は設定・シード・コードバージョン（コミット SHA）を記録する
- 同一入力に対して同一出力が得られることをテストで担保する

### NFR-010 セキュリティ

- 秘密情報をコミットしない
- CI の policy_check で機械的に検査する

### NFR-020 保守性

- 重要ロジックは単体テストを必須とする
- 変更は 1PR で理解可能な粒度に分割する
- コードの識別子は英語、説明文・PR・Issue は日本語で記述する

### NFR-030 フィードバック容易性

- 研究成果は dev-orchestration-template に適用可能な形式で文書化する
- 実験コードと本番品質コードを明確に分離する（experiments/ vs src/）

## 受入条件（全 PR 共通）

| ID | 条件 |
|---|---|
| AC-001 | 変更は要件・ポリシー・計画のいずれかに整合する |
| AC-010 | 必要なテストが追加または更新されている |
| AC-020 | CI（lint / type / test / policy_check）が成功している |
| AC-030 | 変更に応じて正本（docs）が更新されている |
| AC-040 | PR に検証手順と結果が記載されている |
