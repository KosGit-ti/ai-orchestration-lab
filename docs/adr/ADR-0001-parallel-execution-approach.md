# ADR-0001: エージェント並列実行の採用アプローチ

## ステータス

承認

## コンテキスト

現行の dev-orchestration-template パイプラインでは、実装フェーズのエージェントが逐次実行される。
複雑な実装タスク（ML 特徴量エンジニアリング等）では、タスクを独立したサブタスクに分解し、
複数のワーカーで並列実行することで所要時間を短縮し、品質を向上させたい。

具体的な要件として以下が提示された：

- 司令塔（Opus 4.6）＋複数作業者（Sonnet 4.6 / Codex 5.3）の Fan-Out/Fan-In 型並列実行
- 司令塔による品質検証・再指示ループ（パフォーマンス最大化ゲート）
- 異種モデル混成（ヘテロジニアス・マルチモデル）構成
- 並列ワーカー数の可変制御（3〜10、困難なら 5 固定をフォールバック）
- 長時間実行の許容設計

## 検討した選択肢

| 選択肢 | メリット | デメリット |
|---|---|---|
| A1: asyncio + LiteLLM | シンプル、学習コスト低、プロトタイプに最適 | 状態管理・可視化が手動、拡張性に限界 |
| A2: LangGraph Fan-Out/Fan-In | 型安全、Fan-Out が第一級、可視化標準装備、Phase 2 相乗効果 | 学習コスト、API 発展途上 |
| A3: Task Queue (Celery/Redis) | 水平スケーリング、耐障害性 | インフラコスト過大、研究フェーズに不適 |
| A4: A2A プロトコル | 標準プロトコル、言語非依存 | インフラコスト極大、Phase 4 向け |

詳細比較は [設計調査報告](../research/parallel-execution/design-investigation.md) を参照。

## 決定

**段階的アプローチ** を採用する：

1. **PoC（N-002）**: A1（asyncio + モック）で Fan-Out/Fan-In + 品質ゲートパターンの概念実証を行う
2. **本格実装（N-003〜）**: A2（LangGraph）で本格実装する
3. **フォールバック**: LangGraph に問題が生じた場合は A1 で代替する
4. **長期**: A4（A2A）は Phase 4 で再評価する

### 理由

- LangGraph（A2）は Fan-Out/Fan-In が `Send` API で第一級にサポートされており、
  要件への適合度が最も高い
- Pydantic による型安全な状態管理は Phase 2（型安全オーケストレーション）と直接的な相乗効果がある
- ただし、最初のプロトタイプは A1（asyncio）で迅速に検証し、
  パターンの有効性を確認してから LangGraph に移行する方が効率的

## 影響

- `src/orchestration_lab/parallel/` に並列実行フレームワークの型定義を追加する
- `experiments/parallel-execution/` にプロトタイプコードを追加する
- N-003 で LangGraph ベースの実装に移行する際、A1 の型定義はそのまま活用可能
- Phase 2（B-001）の型安全オーケストレーション研究と設計を共有できる
- 外部 API（LiteLLM 経由の LLM 呼び出し）は N-003 で導入する際に ADR を追加で記録する
