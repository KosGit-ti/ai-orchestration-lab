# エージェント並列実行 — 研究ノート

## 概要

AI エージェントパイプラインにおけるエージェント並列実行が実装品質に与える影響を調査・検証する。

## 背景

現行の dev-orchestration-template では、エージェントは基本的に逐次実行される：

```
implementer → test-engineer → CI → auditor-spec → auditor-security → auditor-reliability → ...
```

監査エージェントの並列実行は設計文書で推奨されているが、Copilot Chat のフレームワーク制約により実現が困難である。

## 研究課題

### RQ-1: 並列実行は品質を向上させるか

- **仮説**: implementer と test-engineer を並列に実行（TDD 的アプローチ）すれば、CI 通過率が向上する
- **計測方法**: CI 修正ループ回数、監査指摘数、トークン消費量を比較
- **ベースライン**: 逐次実行の現行データ

### RQ-2: 監査の並列実行は実行時間を短縮するか

- **仮説**: 3つの監査エージェントを真に並列に実行すればパイプライン全体の所要時間が短縮される
- **制約**: Copilot Chat の sub-agent は逐次呼び出ししかサポートしない
- **回避策の候補**:
  - プログラマティック API（LangGraph 等）への移行
  - 複数の Copilot Chat セッションの並行利用
  - A2A プロトコルによるエージェント間非同期通信

### RQ-3: コンテキスト共有の影響

- **仮説**: 並列実行時にコンテキスト（変更ファイル一覧、Serena 分析結果等）を共有する仕組みが無いと品質が低下する
- **検証方法**: コンテキスト共有あり/なしでの品質比較

## 候補アプローチ

### A1: Copilot Chat 内の疑似並列（低コスト）

- Orchestrator が複数のサブエージェントを「同時に」呼び出す指示を出す
- 実際には逐次だが、各エージェントに並列実行を前提とした independent な指示を与える
- メリット: 既存インフラで即座に検証可能
- デメリット: 真の並列ではない

### A2: LangGraph によるプログラマティック並列（中コスト）

- LangGraph のグラフ実行で Fan-Out / Fan-In パターンを実装
- メリット: 真の並列実行、型安全
- デメリット: Copilot Chat から離れるため別環境が必要

### A3: A2A プロトコルによる分散並列（高コスト）

- 各エージェントを A2A エンドポイントとしてデプロイ
- メリット: スケーラブル、標準プロトコル
- デメリット: インフラ構築コストが高い

## 評価指標

| 指標 | 計測方法 | 現行ベースライン |
|---|---|---|
| CI 初回通過率 | CI 修正ループ回数 | 未計測（要データ蓄積） |
| 監査指摘数 | Must / Should 指摘数 | 未計測 |
| パイプライン所要時間 | 開始〜PR 作成の時間 | 未計測 |
| トークン消費量 | エージェントごとの推定消費 | 未計測 |
| コード品質 | テストカバレッジ、型エラー数 | 未計測 |
